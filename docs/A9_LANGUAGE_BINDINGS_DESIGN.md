## 1. Goals and non-goals

### Goals

1. **Expose rgx's value proposition across FFI.** The "Beyond regex: what rgx adds" chapter (book/src/why-rgx.md) identified seven differentiators. Five of them translate cleanly across FFI:
   - Embedded code blocks (Lua / JS / Rhai / WASM run inside rgx; the host language never sees them)
   - `tail_file` / streaming
   - Sandbox modes
   - Structured match events
   - `code_result` (typed values returned from matches)

   Two don't translate well (host-language predicate callbacks and host-language match steering — both cgo-style FFI per-call overhead kills them). The C ABI exposes the five, and provides an *intentionally minimal* surface for the two so users who must use them know what they're paying for.

2. **C ABI as the universal entry point.** A single, stable, documented C header generated by `cbindgen` is the foundation. Per-language wrappers (Go / Python / Julia / Zig / Ruby / PHP / Swift / Kotlin) layer on top via standard C-interop machinery. One implementation effort unlocks ~10 host languages.

3. **PCRE2-compatible behaviour at the API surface.** The Rust `Regex::find_first` / `find_all` / `replace` semantics carry over verbatim; users porting from PCRE2 should see identical match results on every classifier-positive pattern (rgx is at 12,806 / 4 / 0 / 0 against PCRE2's test corpus).

4. **Safe FFI by construction.** No Rust panics cross the FFI boundary (every entry point catches `panic::catch_unwind`); no UAF (opaque pointers with explicit retain/release); no buffer overflows (every byte-buffer API takes an explicit length).

### Non-goals

- **Replacing rgx-core.** The C ABI is a thin shim over the existing Rust API. Internal implementation evolves freely; the C ABI is the only stable contract.
- **Exposing rgx's full Rust surface.** Many internal types (`ExecContext`, `RegexVM`, `LazyDfa`, `TaggedDfa`, etc.) are implementation details. The C ABI exposes only `Regex` + `Match` + the embedded scripting facilities + observers + `tail_file`.
- **Bindings for languages without C interop.** Pure-JVM Kotlin, browser JS, etc. would need a different transport (JNI, WebAssembly export, etc.). Out of scope for A9 first pass.
- **Async I/O bindings on the first pass.** Rust's async ecosystem is unique; a portable async-across-FFI story would need substantial design work. `tail_file`'s synchronous version (`with_blocking_handle`) ships first; the async variant is a follow-up.
- **Per-language idiomatic wrappers.** Phase 1-4 ship the C ABI. Per-language idiomatic wrappers (e.g., "feels like a Python module") are *separate* projects each with their own design doc. A9's contract is the C surface.

Two new artifacts:

1. **`rgx-capi/`** — a new crate in the workspace alongside `rgx-core` and `rgx-cli`. Compiles to both `cdylib` (for runtime linking) and `staticlib` (for embedded). Re-exports the rgx-core Rust API through a stable C ABI via `extern "C"`.
2. **`rgx.h`** — auto-generated by `cbindgen` from `rgx-capi/src/lib.rs`. Distributed alongside the compiled library. Becomes the contract.

Per-language wrappers are *not* part of A9 — they're follow-on projects. A9 ships the foundation only.

## 3. The C API surface

### 3.1 Naming convention

- All public symbols prefixed `rgx_`.
- Snake case (`rgx_find_first`, not `RgxFindFirst`).
- Opaque pointer types: `RgxRegex*`, `RgxMatch*`, `RgxLuaCallback*`, etc. — Pascal-case for the type name, leading `Rgx`.
- Enums: `RGX_OK`, `RGX_ERR_INVALID_PATTERN`, etc. — screaming snake case, leading `RGX_`.

This matches PCRE2 conventions (`pcre2_*` functions, `PCRE2_*` constants) which most regex-API users are already familiar with.

### 3.2 Error model

Every fallible function returns `int32_t`:

- `0` (= `RGX_OK`) on success.
- Negative on failure (`RGX_ERR_*` codes — see §3.3 for enumeration).

Successful results pass through *out parameters* — e.g., a compile-call returns the `RgxRegex*` via `RgxRegex** out_regex`. This convention works in every FFI host (Go, Python, Julia, etc. all handle out-pointers cleanly).

The most-recent error is also stored in **thread-local storage** for callers who want a human-readable diagnostic:

```c
const char* rgx_last_error(void);  // borrowed pointer, valid until next rgx_* call on this thread
```

No Rust panic crosses the boundary. Every entry point wraps its body in `panic::catch_unwind`; a panic surfaces as `RGX_ERR_INTERNAL` with the panic message stored in the thread-local error.

### 3.3 Error codes

| Code | Meaning |
|---|---|
| `RGX_OK` (0) | Success |
| `RGX_ERR_NULL_POINTER` (-1) | A required pointer argument was null |
| `RGX_ERR_INVALID_PATTERN` (-2) | Pattern failed to compile (with `rgx_last_error` giving the syntax-error message) |
| `RGX_ERR_INVALID_UTF8` (-3) | A text input was not valid UTF-8 (only when `RgxRegex` was compiled as a `Regex`, not `BytesRegex`) |
| `RGX_ERR_BUFFER_TOO_SMALL` (-4) | An out-buffer wasn't large enough; the required size is in the corresponding `out_len` parameter |
| `RGX_ERR_LIMIT_EXCEEDED` (-5) | A configured safety limit fired (max_steps / max_backtrack_frames / max_recursion_depth / max_trail_entries) |
| `RGX_ERR_FEATURE_DISABLED` (-6) | The requested feature requires a Cargo feature not enabled in this build (Lua / JS / Rhai / WASM / JIT) |
| `RGX_ERR_INVALID_HANDLE` (-7) | A handle was passed to a function that doesn't accept it (e.g. `rgx_match_free` on an `RgxRegex*`) |
| `RGX_ERR_INTERNAL` (-99) | An unexpected panic from the Rust engine; the panic message is in `rgx_last_error` |

This list is **append-only**. New error conditions get new negative codes; existing codes never change meaning.

### 3.4 Memory model

**Opaque pointers with explicit retain/release.** Every Rust-owned type that crosses the boundary becomes an opaque `*mut T`. The host language owns the pointer's lifetime; the C ABI provides explicit `rgx_*_free` functions.

```c
RgxRegex* re = NULL;
int32_t rc = rgx_compile("(\\d+)", &re);
if (rc != RGX_OK) { ... }

// ... use re ...

rgx_regex_free(re);  // must call exactly once
```

Optionally, reference-counted handles for shared use across threads (`rgx_regex_retain(re)` returns a second handle that must be freed independently). RGX's `Regex` is `Send + Sync` already, so the underlying engine is thread-safe; retain/release just manages the FFI ownership.

**Match results carry borrowed slices.** `RgxMatch*` is an opaque pointer holding the match span + capture positions, but the *original text* is NOT owned by the match — it stays alive in the caller's buffer. The host language must keep the text buffer alive for as long as it accesses `rgx_match_text(m)` etc.

Convention: every function that reads from a text buffer takes the buffer as `(const uint8_t* text, size_t len)`; never as a null-terminated C string. This is mandatory for the bytes-API path and is also more efficient on the str-API path (no `strlen` walk).

### 3.5 Threading

- **`RgxRegex*` is thread-safe for reading.** Multiple threads can call `rgx_find_first(re, ...)` concurrently on the same regex. Mirrors the Rust `Regex` `Send + Sync` story.
- **`rgx_last_error` is per-thread.** Each thread has its own thread-local error string.
- **`RgxMatch*` is single-owner.** Pass between threads is allowed; concurrent access from multiple threads is undefined behaviour. Same model as PCRE2's `pcre2_match_data`.
- **Callbacks (Lua/JS/Rhai/WASM) execute in the caller's thread.** The runtime is single-threaded within each match attempt. Cross-thread callback registration is supported; cross-thread callback *invocation* during an in-flight match is not.
- **`rgx_tail_file` spawns a background thread.** The user-supplied callback is invoked from that thread. The user is responsible for thread-safety of the callback's captured state.

### 3.6 ABI stability

**Stable contract**: function signatures, error codes, struct layouts marked `#[repr(C)]`. Renaming or reordering a function argument is a breaking change.

**Unstable**: opaque-pointer internals (Rust changes freely), the engine implementation (DFA/TDFA/JIT decisions), the runtime perf characteristics.

Versioning: semantic version on the `.so`/`.dylib`. Major bump iff the C ABI signatures change; minor for new functions; patch for bug fixes.

The C header carries `RGX_VERSION_MAJOR` / `RGX_VERSION_MINOR` / `RGX_VERSION_PATCH` constants. `rgx_runtime_version(int* major, int* minor, int* patch)` lets callers check at runtime that the loaded `.so` matches the header they compiled against.

### 3.7 Build artefacts

`rgx-capi/Cargo.toml`:

```toml
[lib]
crate-type = ["cdylib", "staticlib", "rlib"]
```

Build produces:
- `librgx.so` / `librgx.dylib` / `rgx.dll` — dynamic library, for runtime linking.
- `librgx.a` / `rgx.lib` — static library, for embedded use.
- `librgx.rlib` — Rust intermediate, for Cargo consumers (rare but useful for the Rust-side wrapper crate).

`rgx-capi/build.rs` runs `cbindgen` to generate `target/include/rgx.h`. Optionally distributes a `pkg-config` `.pc` file pointing at install locations.

## 4. API surface — phased

The C API ships in four phases. Each is its own commit; later phases depend on earlier ones.

### Phase 1: scaffolding + basic matching

```c
// Lifecycle
int32_t rgx_compile(const char* pattern, size_t pattern_len, RgxRegex** out_regex);
int32_t rgx_compile_bytes(const uint8_t* pattern, size_t pattern_len, RgxRegex** out_regex);
void    rgx_regex_free(RgxRegex* re);
RgxRegex* rgx_regex_retain(RgxRegex* re);

// Diagnostics
const char* rgx_last_error(void);
void rgx_runtime_version(int32_t* major, int32_t* minor, int32_t* patch);

// Basic matching — synchronous, single-match, no captures yet
int32_t rgx_is_match(RgxRegex* re, const uint8_t* text, size_t text_len, int32_t* out_matched);
int32_t rgx_find_first(RgxRegex* re, const uint8_t* text, size_t text_len,
                       size_t* out_start, size_t* out_end);
```

Deliverable: a Rust user can compile, then write a 5-line C program that compiles a pattern, runs `is_match` on a buffer, frees, and exits cleanly. Header generated by `cbindgen`. CI runs the C-side smoke test on Linux/macOS.

### Phase 2: captures + iteration

```c
// Captures
int32_t rgx_find_first_captures(RgxRegex* re, const uint8_t* text, size_t text_len,
                                RgxMatch** out_match);
void    rgx_match_free(RgxMatch* m);

size_t  rgx_match_span(RgxMatch* m, size_t* out_start, size_t* out_end);
size_t  rgx_match_group_count(RgxMatch* m);
int32_t rgx_match_group(RgxMatch* m, size_t group_index,
                        size_t* out_start, size_t* out_end);
int32_t rgx_match_named_group(RgxMatch* m, const char* name, size_t name_len,
                              size_t* out_start, size_t* out_end);

// Iteration — opaque iterator handle
int32_t rgx_find_iter_new(RgxRegex* re, const uint8_t* text, size_t text_len,
                          RgxFindIter** out_iter);
int32_t rgx_find_iter_next(RgxFindIter* iter, RgxMatch** out_match);  // RGX_OK = match, RGX_ERR_END = done
void    rgx_find_iter_free(RgxFindIter* iter);

// Replace
int32_t rgx_replace(RgxRegex* re, const uint8_t* text, size_t text_len,
                    const uint8_t* replacement, size_t replacement_len,
                    uint8_t* out_buf, size_t out_buf_size, size_t* out_required);
int32_t rgx_replace_all(RgxRegex* re, const uint8_t* text, size_t text_len,
                        const uint8_t* replacement, size_t replacement_len,
                        uint8_t* out_buf, size_t out_buf_size, size_t* out_required);
```

Buffer convention for `replace`: caller provides `out_buf` of size `out_buf_size`. If too small, returns `RGX_ERR_BUFFER_TOO_SMALL` with the required size in `*out_required`. Caller retries with a larger buffer.

### Phase 3: configuration + safety limits

```c
// Configuration setters (interior-mutable; thread-safe)
int32_t rgx_regex_set_max_steps(RgxRegex* re, uint64_t limit);  // 0 = unlimited
int32_t rgx_regex_set_max_backtrack_frames(RgxRegex* re, uint64_t limit);
int32_t rgx_regex_set_max_recursion_depth(RgxRegex* re, uint64_t limit);
int32_t rgx_regex_set_max_trail_entries(RgxRegex* re, uint64_t limit);

// Execution mode (Pure / Safe / Full)
int32_t rgx_compile_with_mode(const char* pattern, size_t pattern_len,
                              RgxExecutionMode mode, RgxRegex** out_regex);

// Match semantics
int32_t rgx_regex_set_match_semantics(RgxRegex* re, RgxMatchSemantics semantics);

// Introspection
int32_t rgx_regex_uses_c2(RgxRegex* re, int32_t* out_yes);
int32_t rgx_regex_uses_tdfa(RgxRegex* re, int32_t* out_yes);
int32_t rgx_regex_uses_jit(RgxRegex* re, int32_t* out_yes);
```

### Phase 4: embedded scripting hosts

This is the unique-to-rgx surface. Each embedded host registers via its own function; the actual scripting code lives inside the pattern as `(?{lua:...})` etc.

```c
// Native callbacks (C function pointers)
typedef int32_t (*RgxNativeCallback)(RgxExecContext* ctx, void* user_data);

int32_t rgx_register_native(RgxRegex* re, const char* name, size_t name_len,
                            RgxNativeCallback callback, void* user_data);

// Lua / JS / Rhai / WASM register through the same machinery the Rust API uses;
// the C ABI just exposes lifecycle hooks. The pattern syntax (?{lua:print(MATCH)})
// is the actual scripting surface — these functions just install the runtime.
int32_t rgx_register_wasm_module(RgxRegex* re, const char* name, size_t name_len,
                                  const uint8_t* module_bytes, size_t module_len);

// Variables (host-provided data accessible from (?{...}) blocks)
int32_t rgx_set_variable(RgxRegex* re, const char* name, size_t name_len,
                         const uint8_t* value, size_t value_len);
int32_t rgx_set_variable_json(RgxRegex* re, const char* name, size_t name_len,
                              const char* json, size_t json_len);

// ExecContext accessors (called from inside a native callback)
const uint8_t* rgx_exec_context_text(RgxExecContext* ctx, size_t* out_len);
size_t         rgx_exec_context_position(RgxExecContext* ctx);
const uint8_t* rgx_exec_context_match(RgxExecContext* ctx, size_t* out_len);
const uint8_t* rgx_exec_context_group(RgxExecContext* ctx, size_t group_index,
                                       size_t* out_len);
const uint8_t* rgx_exec_context_named(RgxExecContext* ctx, const char* name,
                                       size_t name_len, size_t* out_len);
```

### Phase 5: observers + structured events (deferred until Phase 4 ships)

```c
typedef int32_t (*RgxEventCallback)(const RgxMatchEvent* event, void* user_data);

int32_t rgx_set_event_observer(RgxRegex* re, RgxEventCallback callback, void* user_data);
int32_t rgx_clear_event_observer(RgxRegex* re);

// RgxMatchEvent is a tagged-union struct (repr(C)) with one variant per event type.
```

### Phase 6: tail_file (deferred until Phase 4 ships)

```c
typedef int32_t (*RgxTailMatchCallback)(RgxMatch* match, void* user_data);

int32_t rgx_tail_file(RgxRegex* re, const char* path, size_t path_len,
                      RgxTailOptions opts,
                      RgxTailMatchCallback callback, void* user_data,
                      RgxTailHandle** out_handle);
int32_t rgx_tail_stop(RgxTailHandle* handle);
void    rgx_tail_handle_free(RgxTailHandle* handle);
```

## 5. Implementation plan

### Phase 0: this design doc

- Land `docs/A9_LANGUAGE_BINDINGS_DESIGN.md` (this file).
- Update BACKLOG A9 to reflect Phase 0 landed.
- Cross-link from `book/src/why-rgx.md`'s "From other languages" section.
- No code lands.

### Phase 1: scaffolding (≈ 3-5 days)

- New crate `rgx-capi/` in the workspace.
- `cdylib` + `staticlib` build targets.
- `cbindgen.toml` + `build.rs` generating `rgx.h`.
- Implement the ~7 Phase-1 functions (compile, free, retain, last_error, version, is_match, find_first).
- `rgx_last_error` thread-local storage via `thread_local!`.
- `panic::catch_unwind` wrapper macro for every entry point.
- C-side smoke test in `rgx-capi/tests/c/`. Build with the system C compiler, link against `librgx.{so,dylib,a}`, run.
- CI workflow: build the C smoke test on `ubuntu-latest` + `macos-latest`.

### Phase 2: captures + iteration (≈ 4-6 days)

- `RgxMatch*` opaque type — owns a `MatchResult` + a borrow of the original input text.
- `RgxFindIter*` opaque iterator handle.
- Replace API with buffer-too-small retry convention.
- C-side test corpus mirroring `c2_pike_differential` but at the C level: compile, call, compare against known answers.

### Phase 3: configuration + safety limits (≈ 2-3 days)

- All four `set_max_*` setters.
- `RgxExecutionMode` enum (`RGX_EXECUTION_MODE_PURE` / `_SAFE` / `_FULL`).
- `RgxMatchSemantics` enum.
- Introspection (`uses_c2` / `uses_tdfa` / `uses_jit`).

### Phase 4: embedded scripting (≈ 1-2 weeks)

- Native callback registration via C function pointer + `void* user_data`.
- `RgxExecContext*` opaque + accessor functions.
- WASM module registration (passes module bytes through to `Regex::register_wasm_module`).
- Variable setters.
- Lua / JS / Rhai don't need new C entry points beyond `rgx_compile_with_mode(MODE_SAFE)` — the actual scripting lives inside the pattern string. Phase 4 just plumbs the registration and execution-context API.

### Phase 5 + 6: observers + tail_file (≈ 1-2 weeks combined)

- Observer callback via C function pointer.
- `RgxMatchEvent` tagged-union struct (`#[repr(C)]`).
- `RgxTailHandle*` opaque background-thread handle.
- `RgxTailOptions` `#[repr(C)]` struct with from-beginning / poll-interval / etc.

### Phase 7: per-language wrapper crates (separate projects, prioritised by demand)

Each language gets its own crate / package. Initial priority order per the `why-rgx.md` analysis:

1. **Go** — strongest gap (RE2 has no PCRE2 compat; large user base). Crate `rgx-go` or pure-Go `cgo` wrapper.
2. **Python** — `regex` package competes; rgx wins on embedded scripting. `rgx-python` via `ctypes` (simpler) or `PyO3` (idiomatic, no FFI overhead — separate Rust-direct wrapper).
3. **Julia** — `ccall` is trivial; scientific-computing users want programmable text transforms.
4. **Zig** — `@cImport`; trivial; small audience but no maintenance burden.
5. **Ruby / PHP** — Oniguruma / preg cover the basics; rgx wins on embedded scripting. Add only if demand signal arrives.

Each language wrapper is its OWN BACKLOG entry, not part of A9 proper. A9 ships the foundation; wrappers ship on demand.

## 6. Correctness gates

### 6.1 Per-phase

Each phase's commit must pass:

- `cargo build -p rgx-capi --release` clean on Linux/macOS/Windows (CI matrix).
- `cargo test -p rgx-capi` (Rust-side wrapper tests).
- C-side smoke / corpus tests via `cmake` or a hand-rolled `Makefile` in `rgx-capi/tests/c/`.
- `cbindgen` regeneration is idempotent (no `git diff` after re-running).
- The existing `rgx-core` test suite (1190+ tests + 12,806 conformance) is **untouched** — the C ABI never modifies `rgx-core`'s public API.

### 6.2 ABI stability check

After each phase: run `cbindgen --crate rgx-capi --output checked-rgx.h` and diff against the committed `rgx.h`. The CI fails if the header changes unexpectedly — every header change must be an explicit commit.

### 6.3 Memory leak gate

Phase 1 ships with a Valgrind / ASan job in CI that runs the C-side smoke test under leak detection. Every subsequent phase extends the C-side test to exercise the new functions; the leak job catches regressions.

### 6.4 Cross-platform validation

CI matrix: `ubuntu-latest` × `macos-latest` × `windows-latest`, on every PR touching `rgx-capi/`. Phase 1's smoke test is the gate.

## 7. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Rust panic crosses FFI boundary → undefined behaviour | Every entry point wrapped in `panic::catch_unwind`; panic surfaces as `RGX_ERR_INTERNAL` |
| Caller forgets `rgx_*_free` → memory leak | Documentation + Valgrind CI job + per-language wrappers handle lifecycle automatically |
| Caller uses `RgxMatch*` after freeing the input text | Documented contract; debug-build assertion via stored pointer-validity-marker |
| ABI accidentally changes (struct layout drift, new arg) | `cbindgen` regeneration is idempotent + CI diff gate |
| Cargo `jit` / `wasm` / `lua` features make the C ABI feel inconsistent across builds | `RGX_ERR_FEATURE_DISABLED` error code; introspection functions report build capabilities |
| Threading bugs in callback dispatch | Phase 4's design pins callbacks to caller-thread; cross-thread invocation is documented as UB |
| Build-system complexity (cbindgen + pkg-config + per-platform install paths) | `rgx-capi/build.rs` keeps it self-contained; CI matrix proves the build on all three OSes |
| A8 publish gate means `rgx-capi` can't go to crates.io until PGEN ships | Doesn't block the artefact itself — users can clone + build. Crate publication waits for A8. |

## 8. Open questions

1. **Static vs dynamic linking default?** For deployment, `librgx.so` (dynamic) is cleaner — one binary, easy patching. For embedding (e.g., a Python wheel), `librgx.a` (static) avoids the runtime-linker dance. Phase 1 ships both; the default for `pkg-config` is dynamic.

2. **Async tail_file from C?** Rust's `async fn` doesn't translate to C. Options: (a) C calls the synchronous variant from a background thread of its own choosing; (b) C provides a callback that fires when a match arrives. Phase 6 punts to (b) — same shape as the existing native-callback model.

3. **Error message localisation?** `rgx_last_error` returns English strings. Internationalisation is out of scope; callers can match on the error code and provide their own translation table.

4. **Should we expose the `MatchSemantics` toggle through the C ABI?** Yes — Phase 3 includes it. The set of LeftmostFirst / LeftmostLongest is small enough that adding it costs nothing and respects users who need POSIX-style matching.

5. **PGO / LTO for `librgx.so`?** Worth considering for the release build. Phase 1 ships without; Phase 8-style optimisation pass can add release-profile tweaks once measured.

6. **Symbol versioning (GLIBC-style `@RGX_1.0`)?** Adds complexity but enables side-by-side major versions. Defer; revisit at version 1.0.
